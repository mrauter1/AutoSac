- 2026-04-09 Verifier: The document is close, but it is not implementation-ready yet. Required writer fixes:
  1. Resolve the contradiction between Section 7.3 and Section 10 on message body exposure. Section 7.3 allows `message_preview` to equal the full normalized `body_text` when the message is short, while Section 10 says delivered Slack messages MUST NOT include full untruncated message bodies. Pick one canonical rule and update both sections so a short public reply is handled unambiguously.
  2. Add explicit Slack-text sanitization rules for all user-derived fields rendered into `text`, at minimum `ticket_title` and `message_preview`. The PRD currently leaves Slack control syntax handling undefined, so one implementation could post raw `<!channel>`, `<@U123>`, or angle-bracket markup from ticket content while another escapes or neutralizes it. Specify the required escaping/neutralization behavior so requesters cannot trigger mentions or Slack markup injection through ticket content.
  3. Define send-time behavior for already-created target rows when the named target is later disabled or removed from `SLACK_TARGETS_JSON`. The PRD covers emission-time routing and global `SLACK_ENABLED = false`, but it does not say whether pending/failed rows for a specific target should still send, pause, or dead-letter when that target now has `enabled = false` or no longer exists. Choose one policy and state the required target-row transition/result.
  4. Tighten the recovery/runtime contract by specifying a finite outbound HTTP timeout and the exact stale-lock recovery requeue state. Right now the PRD says timeouts are retryable and stale locks must become retryable again, but it does not define the required request timeout bound or whether recovered rows become `failed` or `pending` and what happens to `next_attempt_at`. Those choices materially change duplicate-send risk and recovery latency.
